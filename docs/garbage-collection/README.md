|           | 优点                                                                             | 缺点                                                                                                                                                                                        |
| --------- | -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 引用计数  | 简单高效                                                                         | 1. 循环引用导致无法被回收<br> 2. 级联销毁带来的假 STW：当一个庞大的对象树（如巨型链表/图）的根对象引用归零时，会引发连锁的递归释放，可能导致明显的卡顿。                                    |
| 标记-清除 | 解决循环引用                                                                     | 1. 内存碎片问题：清理后导致堆内存出现大量不连续的空间，后续无法分配大对象<br>2. 现代的标记-清除算法中，清除/Sweep阶段往往是可以并发执行的，真正的 STW 主要集中在"标记/Mark"的根节点枚举阶段 |
| 标记-整理 | 解决内存碎片导致无法分配大对象                                                   | 1. STW 时间长：因为需要移动对象，必须更新所有指向该对象的指针，这在传统实现中通常需要较长的暂停时间。<br>2. CPU 开销大：比单纯的清除要多出计算新地址和内存复制的开销。                      |
| 三色标记  | 可并发GC                                                                         | 需结合写屏障避免对象被意外回收                                                                                                                                                              |
| 分代      | 通过分代来频繁扫描新生代对象，降低老年代对象的低效扫描（绝大部分对象会很快死亡） | 实现复杂，需要处理年龄晋升、跨代引用、写屏障。仍然会有STW，并且老年代STW会很重                                                                                                              |

Python GC主要使用引用计数，分代辅助解决循环引用问题。分代垃圾回收只针对可能产生循环引用的容器对象（如 `list`, `dict`, `class` 等），字符串和数字是不参与分代 GC 的。

PHP主要使用引用计数+循环检测。

Go 的协程（Goroutine）极多，对栈的操作极其频繁。如果像 Java/V8 一样在栈上也开启写屏障，会极其影响性能。因此 Go 使用混合写屏障（栈不开启写屏障，只在堆开启）。

Go GC 演进过程：

- Go 1.3：标记-清除（全过程 STW，长达几百 ms 甚至秒级）。
- Go 1.5：引入并发三色标记法 + 插入写屏障。此时 STW 降到了 10-40 ms。但因为栈不开启写屏障，标记结束前必须 STW 重新扫描一次栈（Re-scan），这成了 STW 无法降到 1ms 以下的瓶颈。
- Go 1.8：引入混合写屏障（结合了插入和删除写屏障的优点）。它让堆对象在指针修改时不仅把新引用的对象标灰，也把被覆盖的老对象标灰。这彻底消除了最终阶段重新扫描栈的 STW 需求，成功将 STW 降至 < 1ms。

并发和增量三色标记 GC 几乎一定需要写屏障，否则三色不变性会被破坏。
